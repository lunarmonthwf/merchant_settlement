实际给商家打款的sku项

1.目前商品代金券的成本承担方只能是别样，即使别样运营在商家承担优惠成本的条件下发放的代金券，在系统里记录的成本承担方依旧是别样

对于先发券后付款的商家，目前的解决方案为月度结算时，通过商品代金券的名称搜索使用了指定商品代金券的订单，手动减掉订单中的代金券金额

2.目前SKU_report透出的与结算相关的字段：

MERCHANT_VALUE、R_SALES_TAX_COST、R_MERCHANT_SHIPPING_COST、PROFIT_RATE、SUPPLY_PRICE

结算时仅关注了merchant value，目的是为了merchant value*commission rate，这使得商品价格不包含运费和税费且二者需要参与结算的商家的结算金额有误

3.对于（+）shipping report 、（-）shipping report和（-）refund report，未有整体的关联

当商家在n月发货sku a，sku a会进入n月的（+）shipping report，商家会收到sku a的结算钱款

但是，商家发货不代表用户一定会收到商品，sku a因为某些不可避免的原因，导致用户在n+w月都没有收到货

此时用户要求退款，sku a会进入（n+w）月的（-）refund report。

依据现有的业务操作，若订单a在（n+w）月的refund report 里的订单状态为merchant exception，订单a的结算款项无法追回。

n月

（+）shipping report  sku a

（n+w）月

（-）refund report   sku a

（-）shipping report.  sku a

在结算时，可能不能依据sku订单状态来判断，sku是否要参与结算

场景2:

n月

（+）shipping report  sku a

（n+w）月

（-）refund report   sku a

（-）shipping report.  sku a

（n+w+m）月

当商家在n月发货sku a，sku a会进入n月的（+）shipping report，商家会收到sku a的结算钱款

在（n+w）月时，商家表示实际并没有发货，运营执行删包裹的操作，（n+w）月sku a进入的-shipping report

在（n+w+m）月时，用户表示不想等发货了，要求退款，商家同意退款，运营执行退款操作，（n+w+m）月sku a进入 refund report

依据现有的业务操作，sku a在（n+w）月、（n+w+m）月会分别找商家索要钱款
